# Unit 2: 休暇申請 — 要求仕様 Q&A

## 休暇種別

[Question] 休暇種別は以下で十分ですか？追加があれば教えてください。
  - PAID（有給休暇）
  - SICK（病気休暇）
  - OTHER（その他）
[Answer] PAID（有給休暇）

[Question] 有給休暇の残日数管理は必要ですか？（残数を超えた申請を拒否する等）
  a) 残日数を管理し、超過時はエラーにする
  b) 残日数管理は初期リリースではスコープ外（申請のみ記録）
[Answer] b

## 申請ルール

[Question] 半日休暇（午前休・午後休）は必要ですか？
  a) 必要（半日単位で申請可能）
  b) 不要（1日単位のみ）
[Answer] a

[Question] 申請可能な期間に上限はありますか？（例: 1回の申請で最大30日まで）
[Answer] 現時点ではありません。将来的に残日数管理をする場合は、残日数までと制限します。

[Question] 申請後のキャンセル（取り下げ）は可能ですか？
  a) PENDING 状態なら本人がキャンセル可能
  b) キャンセル不可（管理者に却下してもらう）
[Answer] キャンセル可能

[Question] 同一期間に重複する申請を許可しますか？
  a) 重複不可（既に同じ日に申請がある場合はエラー）
  b) 重複可（管理者が判断する）
[Answer] 重複可 最新情報に上書き

## 承認・却下

[Question] 却下時に理由の入力は必須ですか？
  a) 必須（理由がないと却下できない）
  b) 任意（理由なしでも却下可能）
[Answer] b

[Question] 承認者は任意の ADMIN ユーザーでよいですか？（直属の上司のみ等の制限はない？）
[Answer] 直属の上司

[Question] 承認・却下後に申請者への通知は必要ですか？
  a) メール等で通知する
  b) 画面上で状態を確認できればよい（通知機能は不要）
[Answer] b

## 表示・検索

[Question] 一般社員が自分の休暇一覧を見るとき、フィルタ条件は何が必要ですか？
  a) ステータス（PENDING / APPROVED / REJECTED）のみ
  b) ステータス + 期間指定
  c) 特にフィルタ不要（全件表示）
[Answer]  c

[Question] 管理者画面での休暇申請一覧にページネーションは必要ですか？（50人規模を考慮）
  a) 必要
  b) 不要（全件表示で十分）
[Answer] b

## データ保持

[Question] 却下された申請の保持期間に制限はありますか？（削除ルール等）
  a) 無期限に保持する
  b) 一定期間後に削除する
[Answer] a

## 追加確認

### 承認者の制限

[Question] 「直属の上司」が承認するとのことですが、現在の Employee テーブルに上司の紐付け（manager_id）がありません。どうしますか？
  a) Employee に manager_id を追加し、直属の上司のみ承認可能にする（Unit-0 の変更が必要）
  b) 初期リリースでは任意の ADMIN ユーザーが承認可能とし、上司制限は将来対応とする
[Answer] 所属部署のデータを作成。部署の登録機能をADMINに作成して。
　　　　　また、選択しaのフラグを登録することとする。
　　　　　ユーザ登録画面を作成し、manager_id 登録機能をADMINに作成する。

### 半日休暇の表現

[Question] 半日休暇（午前休・午後休）をどう表現しますか？
  a) 申請に「区分」フィールドを追加する（FULL_DAY / AM_ONLY / PM_ONLY）
  b) 開始日・終了日を DATE ではなく DATETIME にして時間で区別する
  c) その他（具体的に記載）
[Answer] a ただし、英語でなく日本語表記でお願いします。

[Question] 半日休暇の場合、1回の申請で複数日の中に「この日は午前のみ」等の混在は可能ですか？
  a) 不可（1申請は全日 or 半日で統一。混在させたい場合は申請を分ける）
  b) 可能（各日ごとに全日/午前/午後を指定できる）
[Answer] a

### 重複申請時の挙動

[Question] 「重複可・最新情報に上書き」について確認です。同じ日に新しい申請が出た場合の挙動はどれですか？
  a) 既存の PENDING 申請を自動キャンセル（CANCELLED）し、新しい申請を PENDING で作成する
  b) 既存の PENDING 申請の内容（種別・期間・理由）を更新する（申請レコードは1つのまま）
  c) 既存の PENDING 申請がある日にも新規申請可能で、管理者が判断する
[Answer] a 

[Question] 既に APPROVED（承認済み）の日に新たな申請があった場合はどうしますか？
  a) エラーにする（承認済みの取り消しは別フローが必要）
  b) 新規申請を受け付ける（承認されれば上書き）
[Answer] b

## スコープ分割

### 部署・上司管理の実装範囲

[Question] 部署管理・ユーザー登録（manager_id）は Unit-2 に含めますか？それとも別 Unit にしますか？
  a) Unit-2 に含める（休暇申請と一緒に実装する。Unit-2 が大きくなる）
  b) 別 Unit（例: Unit-2a「組織管理」）として先に実装し、Unit-2b「休暇申請」で利用する
  c) Unit-0（基盤）の拡張として先に実装し、Unit-2 は休暇申請のみにする
[Answer] a

[Question] 部署管理に必要な機能はどこまでですか？
  a) 部署の CRUD（作成・参照・更新・削除）すべて
  b) 部署の作成・参照のみ（更新・削除は将来対応）
[Answer] a

[Question] ユーザー登録画面では、以下のどこまでを ADMIN が設定できますか？
  - 社員の基本情報（名前、メール、パスワード）
  - 所属部署
  - 上司（manager_id）
  - ロール（一般社員 / 管理者）
  a) 上記すべて
  b) 基本情報 + 所属部署 + ロールのみ（上司は自動で部署の ADMIN になる）
  c) その他（具体的に記載）
[Answer] a 
　　　　 上司（manager_id）は、上司の氏名で選択する。

### 日本語表記の確認

[Question] 半日休暇の区分フィールドについて「日本語表記」とのことですが、以下のどちらですか？
  a) DB の値は英語（FULL_DAY 等）で保存し、画面表示のみ日本語にする（全日 / 午前休 / 午後休）
  b) DB の値自体を日本語で保存する（「全日」「午前休」「午後休」）
[Answer] a
